# Go-live runbook — the moment the OpenAI key lands

The demo is **already key-independent** (the pre-baked SPSO/NHS fixtures on `/demo` are real
gpt-4o output — no live call on stage). The key is for **(1) live upload quality on a demo call**,
**(2) re-baking the fixtures**, and **(3) the hosted product**. Do these in order; each is quick.

## 1. Local — so you can demo a *live* upload (5 min)
```bash
cd backend
# add to backend/.env (gitignored — never commit the key):
#   OPENAI_API_KEY=sk-...
#   AUTH_SECRET=$(python -c "import secrets;print(secrets.token_urlsafe(48))")
python -m app.admin create-user you@firm.co.uk        # an account to log in with
uvicorn app.main:app --port 8000                       # from backend/, engine on path via repo root
curl http://localhost:8000/health                      # → {"status":"ok","extractor":"openai"}  ← must say openai
```
`app/main.py` now calls `load_dotenv()` on import, so dropping the key in `backend/.env` is enough —
no need to `export`/source it manually before `uvicorn`.
Front end: `frontend/.env.local` already has `NEXT_PUBLIC_API_BASE_URL=http://localhost:8000`;
`npm run dev`, sign in, upload a real tender → confirm **both SPSO disqualifiers** surface and
autofill drafts cite real docs. ⚠️ A fresh 40-pager takes a few minutes on a low-tier key — if you
bought a higher tier this is fast; if not, pre-load the tender before any call.

## 2. Re-bake the demo fixtures — **@generalist owns this** (their lane, the eval + fixtures)
`frontend/src/data/spso-prebake.json` + `nhs-prebake.json` are `GET /requirements`-shaped snapshots
of a real gpt-4o run. Regenerate = run the pipeline against the demo tender with `OPENAI_API_KEY` set,
then overwrite those files. **Watch the throttle:** the old key was 30k TPM and `pipeline._autofill`
is all-or-nothing (one 429 abandons the batch) — a higher-tier key removes this; otherwise throttle
concurrency to 1–2 or ground with the `MockAnswerer`. Full detail: `engine/README.md` §"Demo fixtures + re-baking".
*(Only needed if you want fresher/more tenders in the demo — the current fixtures already work.)*

## 3. Hosted product on Render (for prospects to use after a demo)
`render.yaml` now declares `AUTH_SECRET` (auto-generated, stable) — so tokens sign correctly on deploy.
1. **Render dashboard → the `bidframe-api` service → Environment →** set `OPENAI_API_KEY`. (AUTH_SECRET is
   already generated by the blueprint; `LLM_MODEL=gpt-4o` is set.)
2. **Redeploy** (or just push — it auto-deploys). Then `GET /health` on the Render URL → must say `openai`.
3. **Create accounts** on the instance: Render **Shell** → `cd backend` → `python -m app.admin create-user firm@x.co`.
   (Render's SQLite resets on redeploy, so re-create accounts + re-upload after any redeploy.)
4. **Vercel → the frontend project → Environment →** set `NEXT_PUBLIC_API_BASE_URL` to the Render URL so the
   *hosted* site shows live data (leave unset and it stays the safe mock/`/demo` showcase). Set
   `NEXT_PUBLIC_BOOKING_URL` to your Cal.com link.

## Verify live (the honest gate — do before pointing anyone at it)
- `/health` → `openai` (not `heuristic`).
- Sign in → upload a real tender → **both disqualifiers caught, 0 dangerous misses**.
- Autofill (`Draft my answers` / `?provider=openai`) → grounded answers, **0 bluffs** (every citation real).
- A second account sees **none** of the first's tenders (per-user isolation).

## Don't
- Commit the key (it lives in `backend/.env` locally / the Render dashboard only).
- Point prospects at the Render URL until `/health` says `openai` (heuristic = thin results).
- Live-upload a big tender on a call without a higher-tier key (throttle) — pre-load instead.
